# Qoder Rules 目录

本目录存放 Qoder IDE 的 **Always-On 行为规则**，Agent 在每次会话中自动加载并遵循这些规则。

## 文件清单

| 文件 | 触发方式 | 说明 |
|------|----------|------|
| `think-work-rule.md` | `always_on` | AI 工作行为核心规则集，约束决策权、验证流程、工作效率等 |

## 规则体系概览

### 优先级定义

| 等级 | 含义 |
|------|------|
| `P0` | 不可违反 |
| `P1` | 强烈建议 |
| `P2` | 酌情执行 |

**冲突解决**：P0 > P1 > P2。同级别冲突时，安全相关 > 验证相关 > 效率相关。

### 规则分类

`think-work-rule.md` 包含以下四大类规则：

| 分类 | 编号 | 规则名称 | 优先级 | 核心要求 |
|------|------|----------|--------|----------|
| **一、决策权** | 1 | 用户决策优先 | P0 | 多方案时禁止自行选择，须列出候选由用户决定 |
| | 2 | 理解障碍快速反馈 | P0 | 意图不明时首次回复即列出可能理解或缺失信息 |
| | 3 | 独立评估 | P1 | 对用户方案独立思考合理性，主动指出问题 |
| | 4 | 破坏性操作须确认 | P0 | 删除/覆盖/不可逆操作前必须征得用户确认 |
| **二、验证与反馈** | 5 | 验证闭环 | P0 | 修改文件后必须回读确认，未验证不声称完成 |
| | 6 | 调试根因分析 | P1 | 从错误信息出发分析根因，禁止敷衍式调试 |
| | 7 | 上下文连贯 | P0 | 记住先前讨论和决策，不做矛盾操作 |
| | 8 | 大文件完整读取 | P0 | 超 200 行文件须分段读取或行范围定位 |
| | 9 | 工具执行验证 | P1 | 关键修改追加终端命令验证，简单修改回读即可 |
| | 10 | 失败诚实重试 | P1 | 匹配失败如实告知，最多重试 2 次 |
| **三、编码规范** | — | — | — | （已迁移至 `encoding-constraint` Skill） |
| **四、工作流程** | 11 | AI工作日志维护 | P1 | 每次修改前读取当天日志，修改后追加记录 |
| | 12 | 多文件操作清单 | P1 | 3 个以上文件修改时先用 todo 列清单 |
| | 13 | 任务完整性 | P1 | 多个要求逐一执行确认，不遗漏 |
| | 14 | 分步执行与进度通报 | P1 | 大任务拆分小步骤，单次文件修改不超过 5 个 |
| | 15 | 中断恢复机制 | P1 | 中断后从断点继续，不重复不遗漏 |

## 全局例外

用户可在对话中明确要求跳过某条规则，Agent 遵从但会说明跳过了哪条及潜在风险。

**不可覆盖的底线**（即使跳过规则 4 仍须单独确认）：
- `DROP TABLE` / `DROP DATABASE`
- `git push --force` 到 main/master 分支

## 如何维护

### 新增规则
1. 在对应分类下追加 `### 编号. 规则名称 \`优先级\`` 格式的条目
2. 更新本 README 的规则分类表
3. 检查是否与现有规则冲突或重叠

### 修改规则
1. 直接编辑对应规则内容
2. 如涉及编号变动，同步更新全局例外中的优先级引用和冲突解决说明
3. 更新本 README 的规则分类表
